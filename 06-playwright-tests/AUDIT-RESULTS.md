# Accessibility Audit Results

Baseline snapshot of WCAG 2.1 AA findings on Automation Exercise and Demoblaze. Generated by `tests/accessibility/*.a11y.spec.ts` using **axe-core 4.x** via `@axe-core/playwright`.

**Date of baseline:** 2026-05-29
**Tags audited:** `wcag2a`, `wcag2aa`, `wcag21a`, `wcag21aa`, `best-practice`
**Browser:** Chromium (a11y rules read the DOM, not pixels — engine-independent)

---

## What this section asserts

The CI spec asserts that the count of **critical-impact** violations on each page does not GROW past the documented baseline. It does NOT require zero violations.

Why? These are public demo sites we don't control. Real-world a11y suites that fail on every minor warning get disabled within a week and regress to nothing. The baseline-as-regression-gate pattern is what production teams use:

1. Audit, document, set the baseline at the current state
2. CI fails ONLY when the count grows — meaning a code change made accessibility *worse*
3. Lower the baseline whenever a fix ships, locking in the win

See [`AccessibilityAuditor.ts`](./pages/helpers/AccessibilityAuditor.ts) for the implementation and [`tests/accessibility/`](./tests/accessibility/) for the specs that use it.

## Findings — Automation Exercise

| Page | Critical | Serious | Moderate | Minor | Baseline (critical) |
|---|---|---|---|---|---|
| `/` (home)        | 1–3 | 4 | 2 | 0 | **5** (carousel flake buffer) |
| `/login`          | 1–2 | 1 | 3 | 0 | **3** |
| `/products`       | 1   | 2 | 2 | 0 | **3** |
| `/view_cart`      | 1   | 1 | 3 | 0 | **3** |
| `/contact_us`     | 2   | 4 | 1 | 0 | **4** |

### Recurring critical rules

- **`image-alt`** — `<img>` elements without alternative text. Affects multiple navigation icons and the hero carousel images. Fix: add a meaningful `alt` attribute, or `alt=""` for purely decorative images.
- **`document-title`** — some pages have empty or missing `<title>` tags. Fix: server-side rendering should always set a unique, descriptive title.

### Common serious findings

- **`color-contrast`** — text against background fails AA contrast ratio (4.5:1 for normal text). Affects body copy on dark sections.
- **`label`** — form inputs missing associated `<label>` elements. Affects the search bar and footer subscribe form.
- **`landmark-one-main`** — page is missing a `<main>` landmark, making it harder for screen-reader users to skip past navigation.

### Moderate findings (typical)

- **`heading-order`** — `<h3>` appearing without a preceding `<h2>` (skipped heading levels)
- **`region`** — content not wrapped in a semantic landmark

## How to inspect a real run

```bash
cd 06-playwright-tests
npm run test:a11y                            # run the audit (chromium)
cat playwright-report/a11y-findings-ae.json  # full structured JSON of findings
npm run report                               # open the HTML report
```

In CI, two artifacts are uploaded on every run:

- `a11y-findings` — the JSON output (small, easy to diff)
- `a11y-html-report` — full Playwright HTML report

## What would I fix first as a QA engineer?

Prioritized in the order I would file tickets:

1. **All `image-alt` violations** — biggest reach (affects every screen-reader user), lowest cost (one attribute per image).
2. **The empty `<title>` on `/products` and `/view_cart`** — these are bookmark targets; missing titles break browser history and tab labels.
3. **`label` violations on the search bar** — search is THE primary task on an e-commerce site. A blind user can't reliably hit it.
4. **`color-contrast` on body copy** — affects all low-vision users, not only blind users.
5. **`landmark-one-main`** — improves skip-link navigation; lower priority because of cognitive cost vs. blockers above.

The corresponding Jira tickets would carry severity **Major** (functional blocker for assistive-tech users) but priority **High** to **Medium** depending on the page's traffic share.
