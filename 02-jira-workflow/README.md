## Bug Lifecycle

Full workflow with explicit `Reopened` and `Won't Fix` branches — see [`workflow-diagram.md`](./workflow-diagram.md) for the rendered Mermaid diagram and state-by-state ownership.

```
   Open ──▶ In Progress ──▶ In Review ──▶ Resolved ──▶ Closed
     │                                       ▲           │
     ▼                                       │           ▼
   Won't Fix ──▶ Closed                  Reopened ◀── Reopened
```

### Why explicit `Reopened`?

A bug filed for the first time and a bug whose fix didn't work are very different signals. Lumping reopens back under `Open` loses the **leading indicator of test coverage gaps** that JQL `status CHANGED FROM Closed TO Reopened` gives you. See [`jql-examples.md → Quality Metrics`](./jql-examples.md#4-quality-metrics--trends) for the query.

## Severity vs Priority

These are **independent** axes. A typo in the CEO's name on the homepage is `Minor` severity but `Highest` priority.

| Severity | Technical impact | Example |
|---|---|---|
| **Critical** | System unusable for most users | Checkout fails for all users |
| **Major** | Core flow broken; workaround exists | Discount code not applied |
| **Minor** | Annoying but non-blocking | Tooltip overlaps a button |
| **Trivial** | Cosmetic | Typo in footer |

| Priority | Business urgency | When you assign it |
|---|---|---|
| **Highest** | Drop everything | Live production outage, security disclosure |
| **High** | This sprint | Blocks an upcoming release |
| **Medium** | Next sprint or two | Standard improvement work |
| **Low** | Backlog | Nice to have |

Both fields are present on every bug import in [`bugs-import.csv`](./bugs-import.csv).

## Why I designed it this way

| Choice | Reason |
|---|---|
| Mermaid diagram instead of an image file | Renders inline on GitHub; editable as text; no binary in the repo |
| Pre-built CSV import instead of "click these 5 buttons" | Anyone reviewing the portfolio can replicate the sandbox in 5 minutes, not 30 |
| JQL grouped by *when used* not by *what clause* | The list reads like a QA engineer's actual day; flat dumps look like documentation, not knowledge |
| Real bug tickets sourced from `/01-manual-testing/bug-reports/` | The portfolio is internally consistent — the same bug appears as a Markdown report AND as a Jira ticket |
| Severity ≠ priority distinction made explicit | One of the most-asked QA interview questions; documented here removes ambiguity |
| Screenshots referenced by exact filename | Markdown gracefully degrades; the section is presentable even before the captures arrive |
